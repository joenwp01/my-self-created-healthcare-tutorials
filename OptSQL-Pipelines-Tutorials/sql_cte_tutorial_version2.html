Common Table Expressions

Master CTEs — the named, temporary result sets that make complex SQL readable, maintainable, and powerful. From simple extractions to recursive hierarchies.

Concept

What is a CTE?

A Common Table Expression (CTE) is a named, temporary result set defined at the top of a SQL statement using the WITH keyword. It exists only for the duration of that single query — think of it as giving a name to a subquery so you can reference it cleanly, even multiple times.

Why use CTEs? They break sprawling queries into logical, named steps. Each step is independently readable, testable, and reusable within the same statement. They're the SQL equivalent of well-named functions in application code.

Here's a query without a CTE — a nested subquery that's hard to follow:

Without CTE SELECT department, avg_salary FROM ( SELECT department, AVG(salary) AS avg_salary FROM employees GROUP BY department ) sub WHERE avg_salary > 75000 ORDER BY avg_salary DESC;

Now the same logic rewritten with a CTE:

With CTE WITH dept_averages AS ( SELECT department, AVG(salary) AS avg_salary FROM employees GROUP BY department ) SELECT department, avg_salary FROM dept_averages WHERE avg_salary > 75000 ORDER BY avg_salary DESC;

Both produce identical results, but the CTE version names the intermediate step, making the intent clear at a glance.

Structure

Anatomy of a CTE

WITH cte_name AS ( ← keyword + name + AS -- any valid SELECT ← the CTE body SELECT ... FROM ... ) ← closing parenthesis — no comma if only one CTE SELECT ... ← the "main" (outer) query FROM cte_name ← reference the CTE like a table WHERE ...;

Key rules to remember:

  • WITH appears once at the very start. Multiple CTEs are separated by commas, not additional WITH keywords.
  • A CTE can reference any CTE defined before it in the same WITH block.
  • The main query at the bottom can reference any of the defined CTEs.
  • CTEs are scoped to the single statement — they vanish after the semicolon.
Multiple CTEs syntax: WITH first_cte AS (...),
     second_cte AS (...),
     third_cte AS (...)
SELECT ... FROM third_cte ...;
CTE vs. Temp Table vs. Subquery: A CTE is not materialized (in most engines) — it's essentially an inline view. If you need to reference the same heavy computation across multiple separate statements, consider a temporary table instead. Within a single statement though, CTEs win on clarity.
Worked Example

Your First CTE

Scenario: You have a table orders with columns id, customer_id, amount, order_date. You want to find customers whose total spending exceeds $1,000.

-- Step 1: Define a CTE that totals spending per customer WITH customer_totals AS ( SELECT customer_id, SUM(amount) AS total_spent FROM orders GROUP BY customer_id ) -- Step 2: Query the CTE just like a regular table SELECT customer_id, total_spent FROM customer_totals WHERE total_spent > 1000 ORDER BY total_spent DESC;

What happened:

  • WITH opens the CTE block. customer_totals is the name we chose.
  • Inside the parentheses is a complete, valid SELECT that groups orders and sums amounts.
  • The outer query references customer_totals in its FROM clause as if it were a real table, then filters and sorts.
  • After the semicolon, customer_totals ceases to exist — it's not stored anywhere.
