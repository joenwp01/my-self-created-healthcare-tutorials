Parameterized SQL

Learn to write secure, reusable queries by replacing hard-coded values with parameters — the single most effective defense against SQL injection.

Concept

What is a Parameterized Query?

A parameterized query (also called a prepared statement) separates the SQL structure from the data values. Instead of concatenating user input directly into SQL strings, you insert placeholders and supply the actual values separately.

Why it matters: When you build SQL by concatenating strings, an attacker can inject malicious SQL fragments. Parameterized queries make this impossible because the database engine treats parameter values as data, never as executable code.

Here's the dangerous way — raw string concatenation:

BAD — vulnerable -- Python (UNSAFE!) query = f"SELECT * FROM users WHERE name = '{user_input}'" cursor.execute(query) -- If user_input is: ' OR 1=1 -- -- The query becomes: SELECT * FROM users WHERE name = '' OR 1=1 --' -- This returns EVERY row!

Now the safe way — parameterized:

GOOD — safe -- Python (SAFE) query = "SELECT * FROM users WHERE name = %s" cursor.execute(query, (user_input,)) -- The database treats user_input as a DATA value, -- never as part of the SQL command itself.
Placeholder syntax varies by platform: Python DB-API uses %s, SQLite uses ?, PostgreSQL (psycopg2) supports both %s and named %(name)s, SQL Server / JDBC uses ?, and many ORMs use :name style. In this tutorial we primarily use %s (positional) and :name (named) styles.
Worked Example

Your First Parameterized Query

Scenario: You have a table patients with columns id, name, dob, diagnosis. You need to look up a single patient by their diagnosis value that comes from user input.

-- Step 1: Write the query with a placeholder query = "SELECT id, name, dob FROM patients WHERE diagnosis = %s" -- Step 2: Execute, passing the value in a tuple cursor.execute(query, (user_diagnosis,)) -- Step 3: Fetch results results = cursor.fetchall()

What happened:

  • The %s placeholder tells the database "a value goes here."
  • The second argument to execute() is a tuple containing the actual value. The database engine safely escapes and binds it.
  • Even if user_diagnosis contains malicious SQL, it's treated as a plain string value.
