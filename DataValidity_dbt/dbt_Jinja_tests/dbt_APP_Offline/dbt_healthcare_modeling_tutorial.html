Overview

dbt for Healthcare Data Transformation

dbt (data build tool) transforms raw data already in your warehouse into clean, tested, documented models. Unlike Great Expectations (pre-load validation), dbt operates post-load — structuring raw EHR feeds into layered staging, intermediate, and mart models using pure SQL + Jinja templating.

Sources — Raw Data Entry PointsRaw tables loaded by your EL process (Fivetran, Airbyte, etc.). Defined in YAML with sources:. Accessed in SQL via {{ source('schema', 'table') }}. Provides lineage tracking and freshness monitoring.
Seeds — Static Reference DataCSV files in seeds/ loaded with dbt seed. Perfect for lookup tables: ICD-10 codes, CPT descriptions, facility mappings. Referenced just like models with {{ ref('seed_name') }}.
Models — SQL TransformationsEach model is a SELECT statement in a .sql file. dbt materializes it as a view or table. Models reference other models with {{ ref('model_name') }}, building a DAG (Directed Acyclic Graph) of dependencies.
Tests — Data Quality GuardsSQL queries that return failing rows — a test passes when zero rows are returned. Built-in tests: unique, not_null, accepted_values, relationships. Singular tests go in tests/. Reusable tests use Jinja macros.
Jinja — Dynamic SQL Templatingdbt uses Jinja2: {{ expressions }}, {% statements %}, {# comments #}. Key functions: ref() (models/seeds), source() (raw data), config() (settings).
How this tutorial works: Each exercise simulates a dbt model or test as a SQL query run against live SQLite. You fill in blanks to build the transformation logic, then execute to see results — the same SQL that dbt would compile and run in your warehouse.
