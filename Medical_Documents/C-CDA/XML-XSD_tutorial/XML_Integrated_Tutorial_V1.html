What XML Is and Why It Exists

Markup Concepts

XML — Extensible Markup Language — is a text-based format for storing, transporting, and structuring data so both humans and machines can read it. The W3C standardized XML in 1998 as a simplified subset of SGML, the older ancestor of both XML and HTML.

The core idea of a markup language is annotating data with descriptive labels — tags — that explain what the data represents. Unlike HTML, which has a fixed tag set for page display, XML lets you invent your own tags to describe any data. A tag like <patientId> or <dosageUnit> is self-explanatory to any XML-aware system without documentation.

This self-describing quality is foundational to healthcare data exchange. C-CDA, HL7 v2.x XML, and early FHIR representations are all XML-based. The five XML modules in this tutorial build the foundation; modules 6–10 apply XSD schema validation on top of it.

▶
Worked Demo — Employee Record: Plain Text vs. XML
Same data, two representations
Before — Unstructured Plain Text
Name: Sarah Chen Title: Data Engineer Department: Analytics Hire Date: 2021-03-15 Salary: 95000
After — Self-Describing XML
<?xml version="1.0" encoding="UTF-8"?> <employee> <name>Sarah Chen</name> <title>Data Engineer</title> <department>Analytics</department> <hireDate>2021-03-15</hireDate> <salary currency="USD">95000</salary> </employee>

The plain-text version requires a parser to know field order, delimiters, and date format in advance. Any deviation breaks it. The XML version is self-describing: <name> tells any XML-aware system exactly what follows. The currency="USD" attribute adds metadata without altering structure. The XML declaration identifies the document's XML version and encoding before a single element is read.

02

XML Architecture and Document Structure

SyntaxMarkup

Every XML document has a strict physical structure. Understanding it — and the rules that make a document well-formed — is the foundation for everything that follows, including validation, transformation, and parsing.

An XML document has four structural layers: an optional XML declaration, an optional Document Type Declaration (Module 5), exactly one root element containing everything, and nested content — child elements, attributes, text, and comments.

Well-formedness rules every parser enforces: one root element only; every opening tag has a matching closing tag in the same case; elements must be properly nested (never interleaved); attribute values must be quoted; tag names are case-sensitive. Violating any rule means immediate rejection — no partial parsing.

▶
Worked Demo — Patient Record: Malformed vs. Well-Formed
Three structural errors identified and corrected
Before — Malformed XML (3 Errors)
<patient> <id>PAT-0042</id> <name> <first>Maria</first> <last>Reyes</Last> <!-- wrong case --> </name> <dob>1985-07-22 <status active=true /> <!-- unquoted --> </dob> <!-- bad nesting --> </patient>
After — Well-Formed XML
<patient> <id>PAT-0042</id> <name> <first>Maria</first> <last>Reyes</last> <!-- case matches --> </name> <dob>1985-07-22</dob> <!-- closed first --> <status active="true" /> <!-- quoted --> </patient>

Error 1 — <last> closed as </Last>: XML tag names are case-sensitive; opening and closing must be identical. Error 2 — </dob> appeared after <status>: elements must close in reverse-open order. Error 3 — active=true: all attribute values require quotes.

03

Character Data and the Infoset

DataSyntax

The text between XML tags is character data. XML divides it into two modes: PCDATA (Parsed Character Data, the default) and CDATA sections. In PCDATA mode the parser scans for markup — five characters must be escaped: &lt; for <, &gt; for >, &amp; for &, &apos; for ', &quot; for ". Omitting any causes an immediate parse error.

CDATA sections wrap blocks of literal text the parser skips entirely. They open with <![CDATA[ and close with ]]>. The only forbidden sequence inside is ]]> itself. Use CDATA when content contains many special characters — embedded code, SQL, or clinical free text.

▶
Worked Demo — Clinical Notes: Escaping Special Characters
Unescaped content vs. predefined entities and a CDATA section
Before — Unescaped Content (Parse Error)
<note> <summary> BP: 120/80 & pulse < 100 Patient said "feeling better" O'Brien scale: 4 </summary> </note>
After — Escaped PCDATA + CDATA Section
<note> <summary> BP: 120/80 &amp; pulse &lt; 100 Patient said &quot;feeling better&quot; O&apos;Brien scale: 4 </summary> <rawText><![CDATA[ BP: 120/80 & pulse < 100 Patient said "feeling better" O'Brien scale: 4 ]]></rawText> </note>

<summary> uses predefined entity escapes. <rawText> wraps the same content in a CDATA section — no escaping needed inside it. The parser treats everything between <![CDATA[ and ]]> as literal characters, making it ideal for clinical text, embedded queries, or code snippets.

04

General Entities

Entities

An entity in XML is a named storage unit — a shorthand label that expands to content at parse time. Internal general entities are defined in a DOCTYPE subset with <!ENTITY name "value"> and referenced with &name; anywhere in document content. They create a single maintenance point for repeated text.

External general entities reference content in a separate file using SYSTEM "filename.xml". They were used to modularize large documents but are uncommon today — many parsers disable external entity resolution by default for security reasons.

The five predefined entities from Module 3 (&lt;, &gt;, &amp;, &apos;, &quot;) are built-in general entities — every conforming parser knows them without any declaration. Parameter entities (referenced with %name;) live inside DTDs and are covered in Module 5.

▶
Worked Demo — Prescriptions: Repeated Text vs. Entity References
Defining and using an internal general entity
Before — Repeated Organization Name
<prescriptions> <rx><issuer>Metro Regional Hospital</issuer> <drug>Lisinopril 10mg</drug></rx> <rx><issuer>Metro Regional Hospital</issuer> <drug>Atorvastatin 20mg</drug></rx> <rx><issuer>Metro Regional Hospital</issuer> <drug>Metoprolol 25mg</drug></rx> </prescriptions>
After — Entity Declared and Referenced
<?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE prescriptions [ <!ENTITY mrh "Metro Regional Hospital"> ]> <prescriptions> <rx><issuer>&mrh;</issuer> <drug>Lisinopril 10mg</drug></rx> <rx><issuer>&mrh;</issuer> <drug>Atorvastatin 20mg</drug></rx> <rx><issuer>&mrh;</issuer> <drug>Metoprolol 25mg</drug></rx> </prescriptions>

The internal DTD subset defines entity mrh. Every &mrh; in document content expands to "Metro Regional Hospital" at parse time. If the name changes, only the one declaration needs updating — not every occurrence in the document.

05

DTDs — Historical Context

ValidationHistoryEntities

A Document Type Definition (DTD) is the original XML validation mechanism, inherited from SGML. It defines which elements exist, in what order, with what content models, and which attributes they carry. DTDs predate XSD by several years and are largely superseded — they lack data types, namespace awareness, and fine-grained cardinality. Module 6 picks up exactly where this module ends.

<!ELEMENT name (content-model)> declares an element. Content models: (child1, child2) sequence; (a | b) choice; #PCDATA text; EMPTY; ANY. Quantifiers: + one-or-more, * zero-or-more, ? optional.

<!ATTLIST element attr type default> declares attributes. Types: CDATA, ID, IDREF, NMTOKEN. Defaults: #REQUIRED, #IMPLIED, or a quoted literal. Parameter entities (%name;) live only inside DTDs and allow content-model fragments to be reused across declarations.

▶
Worked Demo — Lab Order: No DTD vs. Internal DTD
Adding a validation grammar to a document
Before — No DTD (Structure Unvalidated)
<?xml version="1.0" encoding="UTF-8"?> <labOrder> <orderId>ORD-2024-001</orderId> <patient id="PAT-042"> <name>James Okafor</name> </patient> <test> <code>CBC</code> <priority>STAT</priority> </test> </labOrder>
After — Internal DTD Defines Structure
<?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE labOrder [ <!ELEMENT labOrder (orderId, patient, test+)> <!ELEMENT orderId (#PCDATA)> <!ELEMENT patient (name)> <!ATTLIST patient id CDATA #REQUIRED> <!ELEMENT name (#PCDATA)> <!ELEMENT test (code, priority)> <!ELEMENT code (#PCDATA)> <!ELEMENT priority (#PCDATA)> ]> <labOrder> ... </labOrder>

The DTD declares labOrder as a sequence of one orderId, one patient, and one-or-more test elements. The id attribute on patient is #REQUIRED. A DTD-validating parser rejects any instance that omits these or adds undeclared elements. DTD validation is a structural layer on top of well-formedness — not a replacement for it. Module 6 shows why XSD was built to replace this.

XSD vs. DTD — Why XSD Exists

SchemaHistory

XML Schema Definition (XSD) was published by the W3C in 2001 to address the fundamental weaknesses of DTDs. Where a DTD can declare that an element exists and what it may contain, it cannot say anything meaningful about what the content is — all text is #PCDATA, whether it's a date, a decimal, an NPI, or free prose. XSD introduces a rich type system, namespace awareness, and fine-grained cardinality constraints — all written in XML itself.

Every XSD document begins with an <xs:schema> root element carrying the W3C namespace declaration: xmlns:xs="http://www.w3.org/2001/XMLSchema". All XSD element and type names are prefixed with xs:. The prefix is conventional; xsd: works equally. Omitting the namespace declaration is the most common beginner error.

CapabilityDTDXSD
Data typesNone — all text is #PCDATA40+ built-in types
Custom constraintsNot supportedxs:restriction with facets

Cardinality+, *, ? onlyminOccurs / maxOccurs (any integer)

▶
Worked Demo — Medication Record: DTD vs. XSD
Same structure in both grammar languages
Before — DTD (No Type System)
<!ELEMENT medication (name, dosage, startDate)> <!ELEMENT name (#PCDATA)> <!ELEMENT dosage (#PCDATA)> ← all text, no types <!ELEMENT startDate (#PCDATA)> ← "hello" would pass <!ATTLIST medication rxId CDATA #REQUIRED> ← no type on attribute
After — XSD (Typed Elements)
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema"> <xs:element name="medication"> <xs:complexType> <xs:sequence> <xs:element name="name" type="xs:string"/> <xs:element name="dosage" type="xs:decimal"/> <xs:element name="startDate" type="xs:date"/> </xs:sequence> <xs:attribute name="rxId" type="xs:string" use="required"/> </xs:complexType> </xs:element> </xs:schema>

dosage typed as xs:decimal rejects <dosage>twice daily</dosage>. startDate typed as xs:date rejects anything not in ISO 8601 YYYY-MM-DD format. The DTD has no such capability — it accepts any string in all three elements. The xs:attribute uses use="required" rather than #REQUIRED and carries an explicit type.

07

Simple Types and Facets

TypesConstraints

XSD's built-in simple types cover the full spectrum of healthcare data. Most commonly used: xs:string, xs:integer, xs:decimal, xs:boolean, xs:date, and xs:dateTime. Two pitfalls appear constantly: xs:number does not exist (use xs:decimal or xs:integer), and xs:dateTime has a capital T — xs:datetime is silently wrong or errors in validators.

When a built-in type is too permissive, narrow it with xs:simpleType and xs:restriction. The base attribute names the type being constrained — it must carry the xs: prefix: base="xs:string", never base="string". Inside the restriction, facets set boundaries: numeric (xs:minInclusive, xs:maxInclusive), string (xs:minLength, xs:maxLength, xs:length, xs:pattern), and enumeration (xs:enumeration).

▶
Worked Demo — Dosage and Age: Permissive vs. Constrained Types
Wrong type names corrected; restriction facets applied
Before — Wrong Types, No Constraints
<xs:element name="dosageMg" type="xs:string"/> <!-- too permissive --> <xs:element name="appointmentDt" type="xs:datetime"/> <!-- wrong case --> <xs:element name="patientAge" type="xs:number"/> <!-- doesn't exist -->
After — Correct Types with Constraints
<xs:element name="dosageMg"> <xs:simpleType> <xs:restriction base="xs:decimal"> <xs:minInclusive value="0.5"/> <xs:maxInclusive value="2000"/> </xs:restriction> </xs:simpleType> </xs:element> <xs:element name="appointmentDt" type="xs:dateTime"/> <!-- capital T --> <xs:element name="patientAge"> <xs:simpleType> <xs:restriction base="xs:integer"> <xs:minInclusive value="0"/> <xs:maxInclusive value="150"/> </xs:restriction> </xs:simpleType> </xs:element>

xs:number does not exist in XSD 1.0. Use xs:decimal for fractional values (dosages, lab values, amounts) and xs:integer for whole numbers (age, counts). The restriction wraps the base type and adds facet bounds. xs:dateTime requires the capital T and expects ISO 8601 combined format: 2024-03-15T09:30:00. Use xs:date for date-only values.

08

Complex Types and Compositors

StructureTypes

When an element contains child elements, it must be given a complex type — defined inline with <xs:complexType> or named and reused. Inside, child elements are grouped by a compositor: xs:sequence — all children must appear in that order; xs:choice — exactly one child must appear; xs:all — all children must appear in any order.

Choosing the wrong compositor is a critical mistake. xs:choice means "pick one" — using it for fields that should all appear means a validator accepts documents missing required data. When all children are required and ordered, use xs:sequence.

Occurrence constraints control repetition. minOccurs defaults to 1; maxOccurs defaults to 1. Set minOccurs="0" for optional elements. For unlimited repetition, use maxOccurs="unbounded" — the word "many" is invalid. maxOccurs must always be greater than or equal to minOccurs.

▶
Worked Demo — Lab Request: Wrong Compositor vs. Correct
xs:choice vs. xs:sequence, and unbounded repetition
Before — Wrong Compositor and maxOccurs
<xs:element name="labRequest"> <xs:complexType> <xs:choice> <!-- allows only ONE element --> <xs:element name="testCode" type="xs:string"/> <xs:element name="patientRef" type="xs:string"/> <xs:element name="orderDate" type="xs:date"/> <xs:element name="diagnosis" type="xs:string" maxOccurs="many"/> </xs:choice> </xs:complexType> </xs:element>
After — xs:sequence, Correct maxOccurs
<xs:element name="labRequest"> <xs:complexType> <xs:sequence> <!-- all must appear, in order --> <xs:element name="testCode" type="xs:string"/> <xs:element name="patientRef" type="xs:string"/> <xs:element name="orderDate" type="xs:date"/> <xs:element name="diagnosis" type="xs:string" minOccurs="0" maxOccurs="unbounded"/> </xs:sequence> </xs:complexType> </xs:element>

A conforming instance under the xs:choice version could supply only testCode and omit the other two required fields — validation would still pass. Under xs:sequence, all four children must appear. maxOccurs="many" is not a keyword; the only valid non-integer value is "unbounded". Adding minOccurs="0" makes diagnosis optional — zero or more occurrences — appropriate when a code may not yet be assigned.

09

Attribute Declarations

AttributesSchema

Attributes in XSD are declared with <xs:attribute>, placed after all compositor content inside a <xs:complexType>. Every xs:attribute requires at minimum a name and a type. Behavior is controlled through use, default, and fixed.

The use property has three valid values: required — the attribute must appear; optional — the attribute may be omitted (the default if use is not specified); prohibited — the attribute must not appear. The value use="true" is invalid and commonly appears from confusion with HTML boolean attributes.

An important mutual-exclusion rule: use="required" and default cannot appear together. A required attribute has no fallback — specifying a default on a required attribute is contradictory and invalid. An attribute with a default is implicitly optional. The fixed property is like default but disallows any value other than the fixed one.

▶
Worked Demo — Prescription Attributes: DTD vs. XSD
ATTLIST declarations translated to xs:attribute with correct use values
Before — DTD ATTLIST
<!ATTLIST prescription status CDATA "pending" <!-- default value --> priority CDATA #REQUIRED <!-- must appear --> source CDATA #IMPLIED> <!-- optional --> <!-- No type validation on any attribute value -->
After — XSD xs:attribute Declarations
<xs:attribute name="status" type="xs:string" default="pending"/> <xs:attribute name="priority" type="xs:string" use="required"/> <xs:attribute name="source" type="xs:string" use="optional"/> <!-- use="optional" is the default — may be omitted -->

default="pending" means: if the instance element omits the status attribute, the XSD processor supplies status="pending" automatically — equivalent to the DTD quoted literal default. This implicitly makes the attribute optional. use="optional" is valid but redundant since it is the default behavior when use is omitted entirely.

Complete Schema Design

SchemaValidation

A production schema combines all techniques from prior modules: a root element with a complex type, nested named and anonymous complex types, correct simple type references, facet-constrained types, compositors that reflect real cardinality, and attribute declarations with appropriate use and default values.

Named complex types are declared at the top level of the schema (direct children of xs:schema) with a name attribute. Any element anywhere in the schema can reference a named type via type="TypeName". This enables reuse — a PatientRefType used in both lab orders and prescriptions is defined once and referenced twice.

The practice exercise is a complete self-contained schema containing errors from three different module domains. Read the entire schema before attempting corrections — some errors affect how surrounding elements validate.

▶

Schematron Fundamentals

Schematron History

Schematron is an ISO/IEC International Standard (19757-3, currently in its 2025 fourth edition) for asserting or reporting the presence or absence of patterns in XML documents. It is classified as a Document Schema Description Language (DSDL) — the same family that includes XSD, but with a fundamentally different purpose: where XSD validates structure and type, Schematron validates business rules and constraints.

A Schematron schema is itself an XML document. Its structure has four nested layers: schema is the root, carrying the namespace declaration xmlns:sch="http://purl.oclc.org/dsdl/schematron". Inside schema, one or more patterns group related rules. Each rule has a context attribute — an XPath expression identifying the node(s) in the instance document the rule applies to. Inside each rule, one or more assert or report elements carry a test attribute — an XPath boolean expression.

Schematron is evaluated by transforming the schema into XSLT (using the open-source SchXslt or skeleton implementation) and applying it to the instance document. The result is an SVRL (Schematron Validation Report Language) document listing all fired assertions and reports. Namespace prefixes used in context and test XPath expressions must be declared on the schema root using sch:ns elements — a frequently missed requirement.

▶
Worked Demo — Minimal Schematron Schema: Broken vs. Correct
The four structural layers and required namespace declaration
Before — Structurally Broken (3 Errors)
<schema> <!-- missing namespace --> <pattern> <rule xpath="//patient"> <!-- wrong attr name --> <assertion <!-- wrong element name --> test="patientId"> Patient must have an ID </assertion> </rule> </pattern> </schema>
After — Correct Schematron Schema
<sch:schema xmlns:sch="http://purl.oclc.org/dsdl/schematron"> <sch:pattern> <sch:rule context="//patient"> <sch:assert test="patientId"> Patient MUST have a patientId element. </sch:assert> </sch:rule> </sch:pattern> </sch:schema>

Error 1 — The root element must carry the Schematron namespace: xmlns:sch="http://purl.oclc.org/dsdl/schematron". Without it, all sch:-prefixed elements are unknown. Error 2 — The sch:rule element uses context as its XPath attribute, not xpath. Error 3 — The assertion element is named sch:assert, not sch:assertion. These three mistakes — wrong namespace, wrong attribute name, wrong element name — account for the majority of beginner Schematron syntax errors.

assert vs. report

Schematron Rules

The most consequential distinction in Schematron is between sch:assert and sch:report. They look identical in syntax — both carry a test XPath expression and a message — but they fire under opposite conditions and carry opposite meanings.

sch:assert fires when its test evaluates to false. It expresses something that must be true — a SHALL constraint. If the condition is not met, validation has failed: the document violates the rule. sch:assert is used for errors.

sch:report fires when its test evaluates to true. It expresses something worth noting when it occurs — an informational finding, a warning, or a SHOULD constraint whose presence is being flagged. sch:report is used for informational findings and non-fatal observations.

Mixing them up is one of the most common Schematron bugs — a schema that uses sch:assert where sch:report is intended will fire errors on valid documents, and a sch:report used for a SHALL constraint will silently pass invalid documents instead of failing them.

▶
Worked Demo — Medication Check: assert and report Used Correctly
SHALL errors vs. informational findings
Before — assert and report Swapped
<sch:pattern> <sch:rule context="//medication"> <!-- fires when true = informational, not error --> <sch:assert test="count(dose) > 1"> NOTE: multiple doses present. </sch:assert> <!-- fires when false = hides missing drug errors --> <sch:report test="not(drugName)"> Medication MUST have a drugName. </sch:report> </sch:rule> </sch:pattern>
After — Correct Usage
<sch:pattern> <sch:rule context="//medication"> <!-- fires when TRUE = informational finding --> <sch:report test="count(dose) > 1"> NOTE: multiple dose elements present. </sch:report> <!-- fires when FALSE = SHALL error --> <sch:assert test="drugName"> Medication MUST have a drugName element. </sch:assert> </sch:rule> </sch:pattern>

In the broken version, the sch:assert fires when count(dose) > 1 is false — meaning it would error on every medication with exactly one dose, which is valid. The sch:report would fire when not(drugName) is true, but only as an informational note — a missing required drug name would silently pass validation.

In the corrected version, sch:report flags the informational condition when multiple doses are present, and sch:assert enforces that drugName must exist — firing an error when it is absent. Notice the test expression also simplified: test="drugName" is truthy when the element exists, so the assert fires (errors) when it is false (missing).

✎

XPath Context and Predicates

XPath Rules

Every Schematron rule has a context attribute — an XPath expression that selects the node(s) the rule applies to. The test attribute on each assert and report is also an XPath expression, evaluated with the context node as the starting point. If either expression is malformed or references an undeclared namespace prefix, the rule silently fails to fire — no error, no warning, just missed validation.

The most common XPath mistakes in Schematron are: using a namespace prefix in context or test without declaring it with a sch:ns element; referencing an attribute without the @ sigil (root selects a child element, @root selects an attribute); and writing path expressions relative to the wrong context node.

For C-CDA and HL7 documents, the cda: namespace prefix must be declared: <sch:ns prefix="cda" uri="urn:hl7-org:v3"/>. Any context or test expression that touches CDA elements must use this prefix. A rule like context="//section" will match nothing in a C-CDA document where every element is in the urn:hl7-org:v3 namespace — it must be context="//cda:section".

▶
Worked Demo — Lab Result Rule: Bad XPath vs. Correct
Namespace prefix declaration, attribute sigil, and context precision
Before — Three XPath Errors
<sch:schema xmlns:sch="http://purl.oclc.org/dsdl/schematron"> <!-- no sch:ns for cda prefix --> <sch:pattern> <sch:rule context="//cda:observation"> <sch:assert test="cda:value/code"> <!-- should be @code --> observation/value MUST have a code attribute. </sch:assert> <sch:assert test="cda:statusCode/code='completed'"> statusCode MUST be completed. </sch:assert> </sch:rule> </sch:pattern> </sch:schema>
After — Namespace Declared, XPath Corrected
<sch:schema xmlns:sch="http://purl.oclc.org/dsdl/schematron"> <sch:ns prefix="cda" uri="urn:hl7-org:v3"/> <sch:pattern> <sch:rule context="//cda:observation"> <sch:assert test="cda:value/@code"> observation/value MUST have a code attribute. </sch:assert> <sch:assert test="cda:statusCode/@code='completed'"> statusCode MUST be completed. </sch:assert> </sch:rule> </sch:pattern> </sch:schema>

Without the sch:ns declaration, the prefix cda: is unknown in XPath evaluation — the rule silently matches nothing. The sch:ns element binds the prefix to the namespace URI and must appear as a direct child of sch:schema, before any patterns.

In the test expressions, cda:value/code selects a child element named code inside value — but code is an attribute in HL7 v3, not a child element. The correct XPath is cda:value/@code. The @ sigil is mandatory when referencing XML attributes in XPath. This is one of the most frequent XPath mistakes in C-CDA Schematron development.

C-CDA Business Rules

C-CDA XPath Rules

In healthcare, Schematron's most important application is validating C-CDA (Consolidated Clinical Document Architecture) documents against conformance rules that XSD cannot express. Most production C-CDA validation failures are Schematron failures, not schema failures. A document can be structurally valid XML, pass XSD validation in full, and still be clinically invalid because a required section is missing, a template ID is wrong, or a code system OID is incorrect.

C-CDA Schematron rules are organized around template IDs — OIDs that identify which template a section or entry claims to conform to. A typical pattern checks: (1) that the template ID is present and has the correct @root value; (2) that required child entries or elements are present; and (3) that vocabulary codes come from the required code system. The XPath path to a template ID is templateId[@root='OID'] — not templateId/@root='OID' which is a predicate expression, not a node test.

The rule severity vocabulary follows HL7 conformance language: SHALL — mandatory, use sch:assert; SHALL NOT — prohibited, use sch:assert test="not(...)"; SHOULD — recommended, typically expressed as sch:report with a note; MAY — optional, informational only. Correct selection of sch:assert vs sch:report is how this vocabulary is implemented in the schema.

▶
Worked Demo — Medication Section: Broken C-CDA Rule vs. Correct
Template ID XPath, code system OID, and SHALL enforcement
Before — Three C-CDA Rule Errors
<sch:schema xmlns:sch="http://purl.oclc.org/dsdl/schematron"> <sch:ns prefix="cda" uri="urn:hl7-org:v3"/> <sch:pattern> <!-- Medication section template: 2.16.840.1.113883.10.20.22.2.1.1 --> <sch:rule context="//cda:section[ cda:templateId/@root= '2.16.840.1.113883.10.20.22.2.1.1']"> <!-- SHALL contain substanceAdministration entries --> <sch:report test="not(cda:entry/cda:substanceAdministration)"> SHALL contain at least one substanceAdministration. </sch:report> <!-- Drug codes SHALL use RxNorm (2.16.840.1.113883.6.88) --> <sch:assert test="cda:entry/cda:substanceAdministration/cda:consumable /cda:manufacturedProduct/cda:manufacturedMaterial /cda:code/@codeSystem='2.16.840.1.113883.6.96'"> Drug code MUST use RxNorm code system. </sch:assert> </sch:rule> </sch:pattern> </sch:schema>
After — All Three Errors Corrected
<sch:schema xmlns:sch="http://purl.oclc.org/dsdl/schematron"> <sch:ns prefix="cda" uri="urn:hl7-org:v3"/> <sch:pattern> <sch:rule context="//cda:section[ cda:templateId[@root= '2.16.840.1.113883.10.20.22.2.1.1']]"> <!-- SHALL: assert fires when condition is false --> <sch:assert test="cda:entry/cda:substanceAdministration"> SHALL contain at least one substanceAdministration. </sch:assert> <!-- RxNorm OID is 2.16.840.1.113883.6.88, not .6.96 --> <sch:assert test="cda:entry/cda:substanceAdministration/cda:consumable /cda:manufacturedProduct/cda:manufacturedMaterial /cda:code/@codeSystem='2.16.840.1.113883.6.88'"> Drug code MUST use RxNorm code system. </sch:assert> </sch:rule> </sch:pattern> </sch:schema>

Error 1 — Template ID predicate syntax: templateId/@root='OID' is an attribute value comparison, not a node selector. In XPath, the correct way to select a templateId element with a specific @root value as a context predicate is templateId[@root='OID'] — the predicate wraps the attribute test in square brackets applied to the element.

Error 2 — assert/report swap: The rule uses sch:report for a SHALL constraint. sch:report fires when its test is true — meaning a missing substanceAdministration (the invalid case, where not(...) is true) would only produce an informational note, not a validation error. Replacing sch:report with sch:assert and simplifying the test to the positive expression enforces the constraint correctly.

Error 3 — Wrong OID: 2.16.840.1.113883.6.96 is the SNOMED CT OID. RxNorm's OID is 2.16.840.1.113883.6.88. OID errors are silent — the schema validates correctly but fires on every valid medication entry, or passes every invalid one, depending on direction.
